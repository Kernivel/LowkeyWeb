ABOUT

UE5 Developer & 3D Artist

I build game systems from the ground up — C++ architectures in Unreal Engine 5, AI behaviors, ability pipelines, Inventory component and networked gameplay — alongside the 3D assets that bring them to life. 3D Modelling, PBR texturing, and game-ready production in Blender and Substance Painter.

My work sits combines tech and art: I implement the systems and then model and texture the assets they depend on.

UE5 SYSTEMS

The weapon system is the backbone of Lowkey's gameplay loop — gunfeel is the first thing a player notices, and the last thing they forget. The architecture is built around that priority: modular meshes for full accessory customisation, and a deliberate split of responsibilities across multiple components — weapon data, runtime stats, recoil, and audio each handled separately. This keeps each layer clean and independently tunable.

VIDEO Weapon System Overview — Placeholder —
[ IMPLEMENTATIONS ]

Weapon data (stats, accessories, ammo count) lives as a lightweight UObject stored in the player's inventory — never spawned as a world Actor. When the player equips a weapon, a separate Actor is created and attached to the character skeleton solely for visual representation. This decoupling means dozens of inventory slots never flood the scene with Actors, and changing stats never requires touching the world mesh.

A dedicated WeaponComponent on the player character acts as a runtime mirror of the currently equipped weapon's stats. When a weapon is equipped, its values are copied into the component. All other systems — UI, GAS abilities, fire logic — read from the component rather than from the weapon data object directly. This keeps the active-weapon query path fast and avoids coupling gameplay systems to inventory internals.

Accessories (suppressors, scopes, grips, stocks, magazines) are data objects attached to the weapon data layer. Swapping an accessory updates stats and signals a visual rebuild on the weapon Actor. This supports any combination of attachments without maintaining a combinatorial set of mesh variants — the visual representation assembles itself from socket-mounted parts at runtime based on what the weapon data holds.

Recoil patterns are authored as curve assets directly in the Unreal Editor — separate curves for camera movement and weapon kick. During fire, the system evaluates these curves in real time, producing patterns whose shape, timing, and recovery can be adjusted without any code changes. This is the core of what drives the tactile, controlled feel targeting the MW2019 gunfeel reference.

COMBAT

NPCs in Lowkey are designed to be readable — predictable enough to reward careful planning, yet dangerous enough to punish mistakes. Their implementation combines Behaviour Trees and GAS Abilities/Attributes. NPCs work with limited information — sight and hearing only — and never have direct access to the player's position. They investigate before committing to combat. In combat, they shoot, reposition actively, take cover to reload, and attempt to flush the player out of cover.

VIDEO Alert State Machine Demo — Placeholder —
[ IMPLEMENTATIONS ]

Detection is modelled as a continuous level that accumulates based on distance — decaying when the player is out of sight. State transitions only occur when this level crosses defined thresholds. An NPC at the edge of perception becomes suspicious; one with a clear line of sight for several seconds escalates to combat. The gradation makes NPCs feel like they're reasoning about partial information.

Reaching 50% detection locks the NPC into Suspicious — decay is suppressed, committing the NPC to investigating rather than resetting. Reaching 100% triggers Combat. Exiting Suspicious resets the level to 25% rather than zero, preserving a baseline tension instead of returning to full calm. Each threshold is defined as a gameplay effect condition, making per-NPC-type tuning a data change rather than a code change.

When entering Suspicious, the NPC stores the location of the detected stimulus and navigates toward it. On arrival, it executes a multi-phase scan — rotating through a wide arc at constant speed, pausing at extremes, then returning. If the player is spotted during this sequence, the state escalates; if the timer expires without detection, the NPC exits and partially resets. The scan angle, speed, and timeout are all configurable per NPC type.

Each alert state maps to a dedicated branch in the Behavior Tree, guarded by a custom decorator that observes the replicated state enum. When the state changes, the decorator aborts the currently running branch and redirects execution to the appropriate one.

GAS — despite its steep initial learning curve — handles replication, abilities, cooldowns, and blocking tags out of the box. In Lowkey, players and NPCs apply buffs and debuffs to each other through the same system. It is a natural fit for this kind of multi-actor interaction, and a great opportunity to build hands-on experience with an industry-standard approach.

The NPC's detection level is a gameplay attribute — the same kind used for health and stamina. This means it replicates automatically, fires change callbacks at thresholds, and can be modified by effects with stacking, duration, and magnitude parameters. Applying detection is architecturally identical to applying damage: parameterised, stackable, and compatible with existing attribute system tooling at no extra implementation cost.

States like currently aiming, reloading, or invisible are expressed as Gameplay Tags applied and removed by abilities. Any system that needs to check a state queries the tag container on the ability system component. This centralises state management, and allows tags to be observed reactively without polling.

Shooting, picking up an item, reloading — all handled through Gameplay Abilities. The player blueprint stays clean — not bloated with custom logic — and adding new abilities requires no changes to the character class.

Animation in Lowkey blends procedural and baked approaches. Weapon sway, camera shake, and recoil motion are procedural — authored as curves and evaluated at runtime, making them reusable across any weapon with only data changes. More deliberate movements — idle stances, ADS transitions, reloads — use baked animations authored per weapon. Player and NPC locomotion use the Motion Matching component from UE5, with retargeted animations from the Unreal Game Animation Sample.

[ IMPLEMENTATIONS ]

Camera and weapon motion during recoil and ADS transitions are driven by curve assets. This means the full shape of the motion — the initial kick, the settle, the recovery arc — is visible and editable in the curve editor without touching code. Camera and weapon meshes each have independent curve sets, allowing the gun to move differently from the camera for a more natural result.

The player character runs two completely separate skeletal instances and animation blueprints — one for the first-person hands and weapon visible only to the local player, and one for the full body visible to other players. This is standard practice: it allows for more visceral first-person animations without making the third-person ones janky. The first-person blueprint focuses on weapon handling and procedural overlays; the third-person blueprint handles locomotion and social gestures.

Character bodies are not fixed skeletal meshes — they are assembled at runtime from modular components by the Mutable plugin. This enables equipment-driven appearance (armour slots, clothing, accessories) without maintaining a combinatorial explosion of mesh variants. The final mesh is generated at runtime based on whatever the character is currently wearing, and updating appearance is a data change rather than a mesh swap.

Weapon rigs are built directly inside Unreal Engine using Control Rig, avoiding the axis-convention issues that come with exporting animated skeletons from Blender. The AG17 rig drives rack, slide, hammer, and magazine release mechanics procedurally through the Control Rig node graph — each part is animated by solver logic rather than hand-keyed, so the motion stays physically plausible across any animation state. Weapon sounds in Lowkey are built from raw gunshot samples that are randomised, pitched, layered, and mixed inside MetaSound graphs to produce variation across fire events.

AUDIO MetaSound Graph Preview — Placeholder —
[ IMPLEMENTATIONS ]

Each weapon uses a custom MetaSound template configured with its own input sounds and parameter values — allowing the same audio graph to produce anything from a lightweight pistol crack to a heavy rifle report without duplicating logic.

Weapon fire is split into multiple independent audio layers: the transient (the sharp crack and mechanical impact), the kick (the Umpf of the weapon hitting the player's body), the mechanics (Giving the weapon a more plastic or metalic sound depending on its components) and the tail (environmental decay and reverb). Separating them through different MetaSound Patches allows greater customisation and layering to get the weapon sound just right.

Since all fire logic originates and is validated on the server, fire events need to reach all connected clients simultaneously. A replication call broadcasts the event to every client at once, and each client triggers its own local audio playback on receipt.

Lowkey follows the server authority model: the server owns all decisions, clients render results. This keeps gameplay deterministic and consistent while keeping network traffic proportional to what players actually need to see.

GIF Multiplayer CO-OP Session — Placeholder —
[ IMPLEMENTATIONS ]

All NPC decision-making — perception, navigation, combat sequencing — runs exclusively on the server. Clients receive only the outputs: position, animation state, and alert level. NPCs have no autonomous behaviour on the client side, eliminating entire categories of desync and ensuring that detection, patrol routes, and combat outcomes are always consistent across all players in the session.

The Gameplay Ability System manages its own replication layer. Attribute changes — health, alert level — propagate to all clients automatically without custom replication code.

3D WORK

All assets produced for Lowkey — weapons, optics, and tactical gear — are built to be both visually convincing and production-sound. Every piece follows the same pipeline: direct-modelled in Blender, baked and textured in Substance Painter, then rigged and animated directly inside Unreal Engine 5. For some of them I have Airsoft references that I own, to helps me get things right. For other I gather as much data as I can on their look and working to reproduce them as best as I can.

[ ABOUT ]

The goal is always assets that hold up both visually and technically — credible surface quality with a polygon count and texture budget suited to a multiplayer session. Hard-surface work is the core focus, but sculpting is part of the pipeline wherever it matters: clothing, fabric, and layered tactical gear are roughed out in Blender or Nomad Sculpt, with subdivision and displacement doing the heavy lifting before baking to the game mesh.

Texturing happens entirely in Substance Painter, built up layer by layer — curvature-driven dirt, procedural wear, cavity shading, and hand-painted detail all converging into a single coherent PBR result. The approach is non-destructive throughout: layer groups stay intact, masks stay editable, and nothing gets flattened until export.

Blender with HardOps and BoxCutter for hard-surface acceleration, and RetopoFlow for clean retopology on organic surfaces. Nomad Sculpt on a Surface Pro for sculpting sessions on the go — especially useful for clothing and fabric before the retopology pass. Substance Painter for PBR texturing, with a library of custom Smart Materials built around specific surface types (military nylon, anodised metal, worn polymer). Unreal Engine 5.7 for rigging, Control Rig setups, and animation — keeping everything in-engine to avoid axis issues from Blender export.

Projects are kept on a shared cloud drive so work moves seamlessly between the desktop workstation and the Surface Pro — desktop for heavy modelling sessions, Surface Pro for sculpting and concept work anywhere.

High-poly and low-poly are produced simultaneously in a single Blender file. The low-poly is direct-modelled to hold its own as a standalone mesh — enough geometry to read cleanly with no visible faceting, correct proportions from every angle. The high-poly is a modifier stack layered on top: Subdivision and Bevels tighten the silhouette and sharpen the edges for a crisper bake. UVs, seams, and Color IDs are set up at this stage while everything is still live.

When ready, the mesh is duplicated into two export objects — _high (modifiers applied) and _low (clean base) — which go straight into Substance Painter for baking. Once textures are complete, rigging and animation are done inside Unreal Engine using Control Rig, eliminating the axis-convention headaches that come with exporting animated skeletons from Blender.
